Java 概念

Java 是一门面向对象的编程语言, 主要应用于网站的后端开发

构成

JDK (Java Development Kit) Java 软件开发工具包

JRE (Java Runtime Environment) Java 运行环境

JVM (Java Virtual Machine) Java 虚拟机, 实际执行 Java 程序的地方

JDK 包含了 JRE, 而 JRE 内置了 JVM

命名规则

变量

  • 名词
  • 允许存在字母, 数字, 下划线(_) 和美元符号($)
  • 不能以数字开头
  • 不能和关键字重名
  • 遵循小驼峰命名规则, 即首字母小写, 随后单词首字母全部大写. 例如 book, bookName, bookCreateTime

类

  • 名词

  • 允许存在字母或数字

  • 遵循大驼峰命名规则, 即首字母大写, 例如 Book, BookName, BookCreateTime

包

  • 名词
  • 允许存在字母或数字
  • 独立的小写单词, 例如 com, raxly, example

数据类型转换

当两个变量的数据类型不一样时, 无法进行运算, 需要转成相同的数据类型时才能进行运算

Java 中的数据类型转换分成了以下两种

  • 隐式转换 : 取值范围小的类型转换为取值范围大的类型, 也称之为自动类型提升

  • 强制转换 : 取值范围大的类型转换为取值范围小的类型

当 int 类型的数字和 double 类型的数字相加, 因为 double 类型取值范围较大, 所以最终结果将自动转成 double 类型

如果需要将 double 类型的数字转成 int 类型, 则需要在前面加上 (int)

取值范围排序

byte < short < int < long < float < double

特殊情况

byte, short, char 这三种数据类型在做运算的时候, 都会先提升为 int, 然后再进行运算

变量提升, 就是在代码执行过程中, JavaScript 引擎把变量声明部分和函数的声明部分提升到代码开头的行为. 变量被提升后, 会被赋予默认值 undefined. 在 ES6 之前, 不管变量声明写在哪里, 最后都会被提升到作用域的顶端

在 ES6 中, 使用 let 或 const 声明变量, 就可以解决上述问题

使用 let 声明或 const 声明变量, 不会造成全局污染, 不存在变量提升, 不允许重复命名

相反, const 在声明时必须赋初始值, 变量的值不可改变

Node.js 是一个开源和跨平台的 JavaScript 运行时环境. 它是几乎任何类型项目的流行工具!

Node.js 在浏览器之外运行 V8 JavaScript 引擎, 它是 Google Chrome 的核心. 这使得 Node.js 非常高效.

Node.js 应用程序在单个进程中运行, 无需为每个请求创建新线程. Node.js 在其标准库中提供了一组异步 I/O 原语, 以防止 JavaScript 代码阻塞, 并且通常, Node.js 中的库是使用非阻塞范例编写的, 这使得阻塞行为成为例外而不是常态.

当 Node.js 执行 I/O 操作时, 如从网络读取\访问数据库或文件系统, Node.js 不会阻塞线程和浪费 CPU 周期等待, 而是会在响应返回时恢复操作.

这允许 Node.js 处理数千个与单个服务器的并发连接, 而不会引入管理线程并发的负担, 这可能是错误的重要来源.

Node.js 具有独特的优势, 因为数百万为浏览器编写 JavaScript 的前端开发人员现在除了客户端代码之外, 还能够编写服务器端代码, 而无需学习完全不同的语言.

在 Node.js 中, 可以毫无问题地使用新的 ECMAScript 标准, 因为您不必等待所有用户更新他们的浏览器——您负责通过更改 Node.js 版本来决定使用哪个 ECMAScript 版本, 您还可以通过运行带有标志的 Node.js 来启用特定的实验性功能.

Node.js 和浏览器之间的区别

浏览器和Node.js都使用 JavaScript 作为其编程语言. 构建在浏览器中运行的应用程序与构建Node.js应用程序完全不同. 尽管它始终是 JavaScript, 但有一些关键的差异使体验截然不同.

从广泛使用 JavaScript 的前端开发人员的角度来看, Node.js应用程序带来了一个巨大的优势: 使用单一语言对所有内容 ( 前端和后端 ) 进行编程的舒适性.

你有一个巨大的机会, 因为我们知道完全\深入地学习一门编程语言是多么困难, 并且通过使用相同的语言在网络上执行你的所有工作 - 无论是在客户端还是在服务器上, 你都处于一个独特的优势地位.

在浏览器中, 大多数时候您正在做的是与 DOM 或其他 Web 平台 API ( 如 Cookie ) 进行交互. 当然, 这些在Node.js中是不存在的. 您没有 document 和 window , 也没有浏览器提供的所有其他对象.

在浏览器中, 我们没有 Node.js 通过其模块提供的所有不错的 API, 例如文件系统访问功能.

另一个很大的区别是, 在Node.js中, 你控制着环境. 除非您正在构建任何人都可以在任何地方部署的开源应用程序, 否则您知道将在哪个版本的Node.js上运行该应用程序. 与浏览器环境相比, 在浏览器环境中, 您无法选择访问者将使用的浏览器, 这非常方便.

这意味着您可以编写Node.js版本支持的所有现代 ES2015+ JavaScript. 由于 JavaScript 发展得如此之快, 但浏览器的升级速度可能有点慢, 有时在 Web 上, 您只能使用较旧的 JavaScript/ECMAScript 版本. 在将代码发送到浏览器之前, 您可以使用 Babel 将代码转换为与 ES5 兼容, 但在 Node.js 中, 您不需要这样做

另一个区别是 Node.js 同时支持 CommonJS 和 ES 模块系统 ( 从 Node.js v12 开始 ), 而在浏览器中, 我们开始看到 ES 模块标准正在实现.

在实践中, 这意味着您可以在 Node.js 中同时使用 require() 和 import, 而只能在浏览器中导入.

V8 JavaScript 引擎

V8 是为 Google Chrome 提供动力的 JavaScript 引擎的名称. 它使用我们的 JavaScript 并在使用 Chrome 浏览时执行它.

V8 是 JavaScript 引擎, 即它解析并执行 JavaScript 代码. DOM 和其他 Web 平台 API ( 它们都构成了运行时环境 ) 由浏览器提供.

最酷的是 JavaScript 引擎独立于它所在的浏览器. 这一关键功能促成了 Node.js 的兴起. 早在2009年, V8 就被选为支持 Node.js 的引擎, 随着 Node.js 越来越受欢迎, V8 成为了支持大量用 JavaScript 编写的服务器端代码的引擎.

Node.js 的生态系统是巨大的, 这要归功于V8, 它也为桌面应用程序提供了动力, 比如 Electron.

Node.js 与 WebAssembly

WebAssembly 是一种高性能的类似汇编的语言, 可以从各种语言编译, 包括 C/C++\Rust 和 AssemblyScript. 目前, Chrome\Firefox\Safari\Edge 和 Node.js 支持它！

WebAssembly 规范详细介绍了两种文件格式, 一种是称为WebAssembly模块的二进制格式, 扩展名为.wsm, 另一种是相应的文本表示, 扩展名称为WebAssembly text 格式, 扩展号为 .wat.

WebAssembly 文档 : https://webassembly.org

关键概念

  • 模块 : 已编译的WebAssembly二进制文件, 即.wasm文件
  • 内存 : 可调整大小的 ArrayBuffer
  • 表 : 未存储在内存中的可调整大小的引用类型数组
  • 实例 : 模块及其内存\表和变量的实例化.

为了使用WebAssembly, 您需要一个.wasm二进制文件和一组API来与WebAssembly通信. Node.js通过全局WebAssembly对象提供了必要的API.

生成 WebAssembly 模块

有多种方法可用于生成WebAssembly二进制文件, 包括:

  • 手工编写WebAssembly ( .wat ), 并使用wabt等工具转换为二进制格式
  • 在C/C++应用程序中使用emscripten
  • 在Rust应用程序中使用wasm包
  • 如果您喜欢类似TypeScript的体验, 请使用AssemblyScript

其中一些工具不仅生成二进制文件, 还生成JavaScript “粘合” 代码和相应的HTML文件, 以便在浏览器中运行.

如何使用它

一旦有了WebAssembly模块, 就可以使用Node.js WebAssembly对象对其进行实例化.

与操作系统交互

WebAssembly模块本身无法直接访问操作系统功能. 第三方工具Wasmtime可用于访问此功能. Wasmtime使用WASI API来访问操作系统功能.

资源

  • WebAssembly常规信息 : https://webassembly.org
  • MDN 文档 : https://developer.mozilla.org/en-US/docs/WebAssembly
  • 手工编写 WebAssembly : https://webassembly.github.io/spec/core/text/index.html

模块规范

CommonJS 为 JavaScript 提供了模块化规范

在模块化规范出现之前, 引入 JS 文件的方法都是通过在 HTML 文件中添加 <script> 标签, 然而这种方法有很多弊端, 例如

  • 无法在 JS 文件内引入其他 JS 文件, 如果需要, 必须有一个 HTML 文件
  • 因为早期的 <script> 标签在 HTML 中是全局的, 因此难以避免一些 JS 全局污染或命名冲突等问题

CommonJS 模块化规范可以很好的解决上述问题, 其具有以下特点

  • CommonJS 允许将功能和逻辑抽离成一个个单独的模块, 使得代码更易于复用和维护
  • 每个模块都有自己独立的模块作用域. 在模块中可以引入其他模块, 同时可以定义允许被其他模块访问的成员
  • 提供了缓存机制, 确保同一个模块只会被加载一次, 从而减少不必要的性能损耗
